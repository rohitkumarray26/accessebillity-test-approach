Windows & Web Accessibility Test Approach

The purpose of this document is to cover the testing approach while performing accessibility testing on Windows/Web platforms.


Accessibility Testing Aspects

  • Keyboard Navigation
  • Screen readers (NVDA, JAWS, and Narrator)
  • High contrast modes (e.g., Aquatic, Desert)
  • Text resize (200%) and reflow at (400%) for web, along with Magnifier for Windows
  • Use the Color Contrast Analyzer (CCA Tool) to verify proper contrast
  • Ensure appropriate text spacing using Stylus
  • Voice Access for Windows to confirm accurate command recognition
  • Automated tools like Accessibility Insights for both web and Windows, and Axe Dev Tools specifically for web, to identify accessibility issues
  • Target Size


1. Keyboard Navigation

Ensure that all interface elements are accessible and operable using only the keyboard, including navigation, activation, and focus management.

  • Use ‘Tab’ and ‘Shift + Tab’ keys to navigate through the interface.
  • Ensure buttons, links, combo boxes, checkboxes, and radio buttons are functional using ‘Enter’ or ‘Space Bar’.
  • Perform ‘Esc’ key functionality for closing dialogs and combo boxes.
  • Check the visibility of the keyboard indicator and design-related concerns.
  • Ensure arrow keys navigate through list items in combo boxes and menu items.
  • Ensure combo boxes are accessible using Alt + down arrow key.
  • Navigate between sections with F6 or ctrl + F6.

WCAG Mapping

Depending on the issue, Keyboard bugs will be mapped in the WCAG rules listed below:

  • WCAG 2.1.1 – Keyboard
  • WCAG 2.4.7 – Focus Visible
  • WCAG 2.4.3 – Focus Order
  • WCAG 1.3.3 – Sensory Characteristics
  • WCAG 2.1.2 – No Keyboard Trap
  • WCAG 2.4.5 – Multiple Ways
  • WCAG 3.2.1 – On Focus
  • WCAG 3.2.2 – On Input
  • WCAG 3.2.3 – Consistent Navigation
  • WCAG 3.2.4 – Consistent Identification
  • WCAG 1.4.13 – Content on Hover or Focus


2. Screen Reader

Verify that screen readers like NVDA, JAWS, and Narrator can accurately announce the name, role, and value of controls, and that navigation is logical.

Changing Modes:

  • NVDA (Version: 2024.3): Insert/Caps Lock + Space Bar
  • Narrator: Insert/Caps Lock + Space Bar
  • JAWS (Version 2024.2406.121): Insert/Caps Lock +Z
    • Verify semantics are announced properly for the controls.
    • Heading: Use the H shortcut to verify the sequential design of headings (e.g., h1 → h2 → h3 → h4 → h5 → h6).
    • List: Use the L shortcut to check list positioning.
    • List Items: Use I shortcut for list items.
    • Landmarks: Use the D shortcut to verify landmarks in NVDA and Narrator, and R for JAWS.
    • Finding Elements:
      • Links: K
      • Buttons: B
      • Graphics: G
      • Table: T
      • Forms: F
      • Element’s list: Insert + F7

    WCAG Mapping

    Depending on the issue, Screen Reader bugs will be mapped to the following WCAG rules:

    • WCAG 1.3.1 – Info and Relationships
    • WCAG 4.1.2 – Name, Role, Value
    • WCAG 1.3.2 – Meaningful Sequence
    • WCAG 4.1.3 – Status Messages
    • WCAG 4.2.9 – Execution of Available Actions
    • WCAG 4.2.10 – Tracking of Focus and Selection Attributes
    • WCAG 3.1.1 – Language of Page
    • WCAG 3.1.2 – Language of Parts
    • WCAG 1.1.1 – Non-text Content
    • WCAG 2.4.2 – Page Titled
    • WCAG 1.4.5 – Images of Text


3. High Contrast Mode

Prerequisites

Navigate to Settings → Accessibility → Contrast themes and select Aquatic or Desert mode.

  • Evaluate the web page in High Contrast mode to ensure it meets the needs of low-vision or color-blind users.
  • Confirm that controls are displayed appropriately, for example:
    • Links appear in blue.
    • Buttons appear in grey.
  • If necessary, use the Color Contrast Analyzer (CCA) to check focus indicators or control visibility in High Contrast modes.
  • If an issue occurs in a particular mode (e.g., Aquatic or Desert), cross-verify in Night Sky and Dusk modes.
  • Verify the CCA for controls and text in both Windows High Contrast mode and within application High Contrast settings.

WCAG Mapping

The WCAG rule for High Contrast is as follows:

  • WCAG 4.3.1 – No Disruption of Accessibility Features


4. Resize/ Reflow Mode

Text Resize and Reflow Evaluation

Ensure that text can be resized up to 200% and reflowed at 400% without any loss of content or functionality. Additionally, verify that Windows applications work seamlessly with the Magnifier tool.

  • Prerequisites for Resize/Reflow for Web:
    1. Set Display resolution to 1280 × 1024.
    2. Set Scaling to 100%.
    3. Zoom the Browser to 200% or 400%.
  • For Windows:
    • Set Resize to 1280 × 768 for general use.
    • Set Reflow to 1024 × 768 for testing reflow.

  • Ensure that all controls on the page remain visible and functional without truncation, overlapping, or any content loss.

  • Magnifier (Windows): Verify the Magnifier functionality using both mouse and keyboard (with Tab keys) across all Windows applications. The default magnification level should be set at 200%.
    • Log all issues under Microsoft Accessibility Standards (WCAG) 4.3.1.

WCAG Mapping

The WCAG rules for Resize/Reflow are listed below:

  • WCAG 1.4.4 – Resize Texts
  • WCAG 1.4.10 – Reflow


5. Text Spacing

Use Stylus tool to ensure that text is properly spaced, avoiding truncation, overlapping, or loss of content.

WCAG Mapping

  • WCAG 1.4.12 – Text Spacing

8. Manual Color Analysis

Target Size Evaluation

For testing Target Size issues on the web, use the CodePen - Target Size Bookmarklet.

  • For Windows applications, no automation tool directly addresses 'target size' issues. However, Accessibility Insights can be used to manually inspect the 'Local size' property for each selected element.

The size of the target for pointer inputs should meet a minimum of 24 by 24 CSS pixels, except in the following cases:

  • Spacing: The target offset is at least 24 CSS pixels from any adjacent target.
  • Equivalent: An alternative control on the same page meets the size criterion.
  • Inline: The target appears within a sentence or block of text.
  • User Agent Control: The target size is defined by the user agent, not modified by the author.
  • Essential: The presentation is necessary or legally required to convey the intended information.

WCAG Mapping

The WCAG rule for Target Size is as follows:

  • WCAG 2.5.8 – Target Size (Minimum)


9. Target Size (Minimum)

For testing Target Size issues on the web, use the CodePen - Target Size Bookmarklet.

For Windows applications, no automation tool directly supports 'Target Size' issues. However, you can use Accessibility Insights to manually inspect the Local size property for the selected element.

Target Size Requirements

The size of the target for pointer inputs should be at least 24 by 24 CSS pixels, except in the following cases:

  • Spacing: The target offset is at least 24 CSS pixels from every adjacent target.
  • Equivalent: The function can be achieved through a different control on the same page that meets the size criterion.
  • Inline: The target is within a sentence or block of text.
  • User Agent Control: The size of the target is determined by the user agent and not modified by the author.
  • Essential: A specific presentation of the target is essential or legally required to convey the intended information.

WCAG Mapping

The applicable WCAG rule for Target Size is:

  • WCAG 2.5.8 – Target Size (Minimum)

Mac Accessibility Test Approach


Accessibility Testing Approach on macOS

The purpose of this document is to outline the testing approach for accessibility testing on macOS. The following key accessibility testing aspects are covered:

  • Keyboard Navigation: Ensuring that all interactive elements can be accessed and operated using a keyboard.
  • Resize: Verifying that content and controls are resizable without loss of functionality.
  • Reflow: Testing that content reflows correctly when resized or zoomed in.
  • Screen Reader (VoiceOver): Verifying that the VoiceOver screen reader announces elements correctly and in sequence.
  • Luminosity Ratio (Colour Contrast Analyzer): Ensuring sufficient contrast ratios for text and backgrounds.
  • Increase Contrast: Ensuring that the application adapts well when the system’s increase contrast feature is enabled.
  • Target Size: Verifying that interactive elements have a minimum target size of 24px by 24px for pointer inputs.
  • Invert Colour (Smart Invert): Checking how the application behaves when the system’s color inversion feature is enabled.
  • Voice Control: Testing the functionality of the application when voice commands are used to interact with it.
  • Text Spacing: Ensuring that increasing text spacing does not cause content overlap or break functionality.
  • Magnification (Zoom): Verifying that the application works well with macOS's magnification/zoom feature.
  • Hover Text for Icons: Ensuring that icons display descriptive hover text (tooltips) for screen readers and visual users alike.

The following are detailed accessibility testing aspects for macOS, covering **Keyboard**, **Resize**, and **Reflow**:

1. Screen Reader: TalkBack

TalkBack Verbosity: High (default)


How to turn on TalkBack on an Android device:

  1. Using the Settings Application: Path: Settings > Accessibility > TalkBack > Toggle On
  2. If Shortcut is enabled:
    • Press and hold both volume keys (Volume Up and Volume Down) simultaneously for about 3 seconds.
    • Press and hold the Power button.

Ensure that TalkBack accurately announces all required information while navigating on interactive elements (buttons, links, images) and text content. Ensure that navigation is logical.


Additional Features Evaluated with TalkBack:

Navigation with Keyboard + Screen Reader

Bluetooth Keyboard + TalkBack: This method involves using a Bluetooth keyboard with TalkBack. Ensure all the content present on the screen is accessible while navigating with Bluetooth Keyboard + TalkBack and that TalkBack is announcing the correct information.

Navigation Keys for TalkBack using Bluetooth keyboard:
  • Navigate to next item: Alt + right arrow
  • Navigate to previous item: Alt + left arrow
  • Activate any control: Alt + enter
  • Go back: Alt + backspace
  • Navigate to first item: Alt + ctrl + left arrow
  • Navigate to last item: Alt + ctrl + right arrow
  • Touch and hold the focused element: Alt + shift + enter

Reference Link: Use TalkBack keyboard shortcuts - Android Accessibility Help

Reading Control

How to cycle among different reading modes:

  • With TalkBack: Swipe Right/Left with three fingers to select a reading mode.
  • Use swipe up / down gestures with one finger to navigate in the selected reading mode.
  • With TalkBack + Bluetooth Keyboard: Use Alt + Shift + Ctrl + Up/Down arrow keys to select a reading mode.
  • Use Alt + Shift + Up/Down arrow keys to navigate in the selected reading mode.

Different types of Reading controls:

  • Characters: Read by character (Swipe down for next, up for previous).
  • Words: Read by word (Swipe down for next, up for previous).
  • Lines: Read by line (Swipe down for next, up for previous).
  • Paragraphs: Read by paragraph (Swipe down for next, up for previous).
  • Headings: Jump between headings.
  • Controls: Jump between interactive elements (buttons, text fields).
  • Links: Jump between links.
  • Speech Rate: Adjust TalkBack speed (Swipe up to increase, down to decrease).
  • Spoken Language: Switch text-to-speech language (if available).

Reference Website: Use the TalkBack menu & reading controls - Android Accessibility Help

Touch Exploration

This is not used as the default test matrix. When we encounter any issue related to a control not accessible via swipe gesture, we verify using Touch Exploration and mention its behavior in the observation.

How to test using Touch Exploration: Touch the screen with your finger and keep it there. As you drag your finger around, TalkBack will announce the items or elements that your finger is currently over.

Prerequisites: Settings > Accessibility > TalkBack > Toggle On.

9. Target Size (Minimum)

Tool: Accessibility Scanner

Use Accessibility Scanner tool to verify Touch Target Size issues.

Ensure all interactive elements on the screen are large enough to be easily tapped or activated without accidental errors.

Exceptions:

  • The target is available through an equivalent control that meets the size requirement.
  • The target is inline (e.g., a link in a sentence).
  • The specific size is essential for the presentation of the content.
  • Spacing Between Elements: Ensure there is adequate spacing around targets to prevent accidental activation of nearby elements, providing a clearer touchable area.

Prerequisite: Change to 24px from Accessibility Scanner settings.

Path to change the setting: Open Accessibility Scanner application > Activate ‘More Option’ button > Select ‘Settings’ > Click Target Size and give input as 24.

WCAG Mapping: WCAG rule for the Target Size is listed below.

  • WCAG 2.5.8 - Target Size (Minimum)

ios Accessibility Test Approach


We Address below accessibility testing aspects in iOS:

  • Screen Reader (Voiceover)
  • Keyboard Navigation (Bluetooth Keyboard)
  • Screen Zoom
  • High Contrast
  • Color Contrast
  • Orientation
  • Voice Control
  • Target Size

1. Screen Reader (Voiceover)

Voiceover is used to navigate on screen contents (Buttons, links, headings, etc.) and verify that information related to content/controls is properly announced while navigating with swipe gestures. We also verify the issues related to the Voiceover focus order, ensuring it lands on all the controls.

Additional Settings Used with Voiceover:

a. Verbosity

Testing is performed at the default verbosity level.
Path: Go to the Settings app, then navigate to Accessibility > Voiceover > Speech > Verbosity.

b. Rotor

To use the Rotor with Voiceover on an iPhone, place two fingers on the screen and rotate them clockwise. Select a Rotor option like "Headings" or "Links," then swipe up or down to navigate through items based on the selected option.

To configure the Rotor, go to the Settings app, then navigate to Accessibility > Voiceover > Rotor. Select the desired navigation options to include in the Rotor menu and adjust them as needed.

c. Audio Ducking

By default, it is turned ON. If any announcement-related issues are encountered, we check the behavior with Audio Ducking turned on and mention the same in the observation.
Path to activate it: Go to the Settings app, then navigate to Accessibility > Voiceover > Audio > ‘Audio Ducking’ switch on.

d. Screen Recognition

Path to activate it: Go to the Settings app, then navigate to Accessibility > Voiceover > Screen Recognition.

By default, it is turned off. If any Voiceover issues such as name, role, state, status message, text-content, or dialog/pop-up alert information are not announced, the behavior with Screen Recognition is mentioned in the observation.

Reference/Source Link: Turn on and practice VoiceOver on iPhone

WCAG Mapping for Screen Reader (Voiceover) Issues

  • WCAG 1.1.1 – Non-text Content

7. Voice Control

We perform voice control testing to verify that contents are accessible with voice commands. Voice control allows users to navigate and control their device using voice commands, which improves accessibility for individuals with mobility impairments and enhances productivity for all users.

Basic Commands to Use Voice Control:

  • Show/Hide Numbers: Say "Show numbers" to activate item numbers, then speak the number you wish to select. Say "Hide numbers" to turn off item numbers.
  • Show/Hide Names: Say "Show names" to activate item names, then say "Click" followed by the name to select an item. Say "Hide names" to turn off item names.
  • Punctuation, Symbols, and Emojis: Say the name of the punctuation mark, symbol, or emoji you want (e.g., "question mark," "percent sign," "happy emoji").
  • Text Navigation and Selection: Use commands like “Move up two sentences,” “Move forward one paragraph,” “Select previous word,” or “Select next paragraph” to navigate and select text.
  • Text Formatting: Use commands like “Bold that,” “Capitalize that,” or “numeral” to format your next phrase as a number.
  • Deleting Text: Use commands such as “Delete that” to remove the last thing you typed or “Delete all” to erase everything and start over.
  • Grid Overlays: Say "Show grid" to display a numbered grid on your screen, or "Show window grid" to limit the grid to the active window. Say a grid number to subdivide the area further.

Steps to Activate Voice Control on iPhone:

Path: Go to the Settings application → "Accessibility" control → Accessibility screen → Toggle the Voice Control button.

For a detailed list of commands, please refer to the official documentation: Use Voice Control on your iPhone

WCAG Mapping for Voice Control:

  • WCAG 4.2.9 – Execution of Available Actions
  • WCAG 4.3.1 – No Disruption of Accessibility Features


8. Target Size

To check target size issues, we use a screen ruler app developed by Info Enum software. We manually verify the issue by taking screenshots and checking the size of the icons, text content, etc., which should be equal to 24px * 24px (length and width), using the horizontal and vertical ruler present in the application.

Steps to Check Target Size on the Scaler Application:

  1. Download: Install the Screen Ruler app from the App Store on apple.com.
  2. Launch: Open the app and provide all the necessary permissions. The main interface will appear.
  3. Import: Capture a screenshot of your screen and upload it into the Screen Ruler app.
  4. Measure: Use the app's dual scales to evaluate the target size in your screenshot.
  5. Adjust: Zoom in on the screenshot, and the ruler will automatically adjust to the zoom level.
  6. Save: After verifying the measurements, take another screenshot to save your results.

Exceptions of Target Size:

  • Spacing: For undersized targets (less than 24px by 24px), they should be arranged in such a way that a 24px diameter circle does not overlap with other targets.
  • Equivalent: If a target is smaller than 24px by 24px but there's another control fulfilling the same function, it can be exempted under the 'Equivalent' exception.
  • Inline: This criterion does not apply to inline targets within sentences or where target size is constrained by line height.
  • Essential: The size and spacing of targets are critical for conveying information, such as in digital maps or interactive data visualizations.

Reference/Source Link: Scaler

WCAG Mapping for Target Size:

  • WCAG 2.5.8 – Target Size (Minimum)
